# GB0170

## Graph-Break Type
*Short name describing what triggered the graph break*

Data-dependent branching

## Additional Information

<!-- ADDITIONAL INFORMATION START - Add custom information below this line -->

This graph break happens whenever a Python `if`, `while`, `assert`, or other
boolean test depends on the contents of a tensor. Which branch runs would depend
on the tensor's values, and `torch.compile` builds one fixed compiled function
before those values are known, so it cannot pick the branch ahead of time.

The obvious form is something like `if x.sum() > 0:`, but the same break shows up
in several less obvious ways. The common thread is that a tensor ends up in a
boolean context, even when the code does not look like it is branching on a
tensor.

### `.any()` / `.all()` look like Python bools, but are not

`tensor.any()` and `tensor.all()` return a one-element boolean tensor, not a
Python `bool`, so testing one in an `if` triggers the break. A very common place
this appears is a NaN/inf guard in a training loop:
```python
import torch


@torch.compile(fullgraph=True)
def fn(x):
    if torch.isnan(x).any():  # data-dependent branch
        return torch.zeros_like(x)
    return x * 2


fn(torch.randn(8))
```

Other everyday examples that hit the same break: a threshold or accuracy check
such as `if (preds == labels).float().mean() > 0.9:`, and a `while` loop whose
condition is a tensor, such as `while residual.abs().max() > tol:`.

### How to fix it

If both branches are cheap tensor expressions, compute them without branching
using `torch.where`, which selects per element (or broadcasts a single boolean):
```python
import torch


@torch.compile(fullgraph=True)
def fn(x):
    return torch.where((x > 0).float().mean() > 0.5, x + 1, x - 1)


fn(torch.randn(8))
```

For common patterns there is often a dedicated branchless op. The NaN guard
above, for example, can be written with `torch.nan_to_num`:
```python
import torch


@torch.compile(fullgraph=True)
def fn(x):
    return torch.nan_to_num(x) * 2


fn(torch.randn(8))
```

If you genuinely need different computation in each branch, use `torch.cond`,
which captures both branches into the compiled function and picks one at run
time:
```python
import torch


@torch.compile(fullgraph=True)
def fn(x):
    return torch.cond(x.sum() > 0, lambda t: t + 1, lambda t: t - 1, (x,))


fn(torch.randn(8))
```

Finally, if the check runs rarely (for example a NaN guard that almost never
fires), it may be simplest to accept the graph break: drop `fullgraph=True` and
`torch.compile` will compile the tensor code around the branch, leaving the
branch itself in regular Python.
